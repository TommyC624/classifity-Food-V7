# CLIP ViT-B/32 智能食物辨識（純 HTML）

這個網頁使用 Transformers.js 和 `Xenova/clip-vit-base-patch32` 進行零樣本影像分類，
毋須 Python 後端，也毋須自行訓練模型。

最新版 Word 清單的 156 項資料已直接內置於 `index.html`。其中155個可從外觀辨認的
食品會在同一次 CLIP 運算中直接比較，避免第一層分類錯誤後停止。網頁會顯示三個中文
候選結果及真正可按的確認按鈕；使用者確認後，結果也會在瀏覽器中保存。隔夜餸應按
存放時間判斷，不會交由相片模型猜測。

網頁包含三個獨立功能：

1. 食物辨認相機：只拍攝食物，使用 CLIP 比較內置清單。
2. 包裝日期相機：另外拍攝日期，使用 Tesseract.js OCR；支援 OCR 候選確認、日期種類、
   重新拍攝及手動日期。
3. 隔夜餸記錄：記錄放入雪櫃的日期和時間，跨過日期後顯示隔夜餸提示。

日期相機會裁剪畫面中央的日期框、轉成灰階、提高對比度及調整過大相片尺寸。支援
`16/08/2026`、`2026-08-16`、`16.08.2026`、`20260816`、`16 AUG 2026`
及 `AUG 16 2026` 等格式，也支援食品包裝常見的兩位年份，例如 `31-07-26`
會轉換成2026年7月31日。

日期OCR v6改用可重用的Tesseract worker，限定只辨認數字及日期符號，並以稀疏文字
模式依次掃描「中央灰階」、「中央黑白」及「完整照片」。若仍失敗，頁面會顯示OCR
實際讀到的內容，方便判斷是裁剪問題、文字問題或日期格式問題。

日期OCR v7加入`Xenova/trocr-small-printed`後備模型。系統先執行三次Tesseract掃描；
只有全部失敗時才透過Transformers.js下載量化TrOCR模型，再辨認中央日期框。完成後
會釋放TrOCR模型，減少iPhone瀏覽器的記憶體佔用。第一次啟動後備模型需要網絡及較長
下載時間。

日期OCR v8再加入上傳版本所用的全圖高對比黑白化方式：亮度按1.8倍提高對比度，
再以140作黑白分界。Tesseract現在依次嘗試「中央灰階」、「中央黑白」、「完整灰階」
及「完整高對比黑白」四個畫面，才啟動TrOCR。原上傳檔遺漏的Tesseract.js載入及錯誤
相機元素ID不會帶入本版本。

v9修正iPhone Safari相機啟動問題。食物相機及日期相機不再只依靠縮成1×1像素的透明
檔案輸入，而會顯示可直接點擊的原生相機輸入；「重新拍攝日期」亦改用HTML原生
`label for`連結，不再用JavaScript模擬點擊。GitHub Pages必須使用HTTPS，並建議直接
在Safari分頁開啟，不要在WhatsApp等App的內置瀏覽器中操作。

v10把TrOCR後備模型換成官方瀏覽器版PP-OCRv5 Mobile。Tesseract四次掃描全部失敗後，
系統才動態載入PaddleOCR.js、PP-OCRv5手機文字偵測模型及文字辨認模型。它會先掃描
原始日期照片，再掃描高對比黑白照片；兩次結果都會交給同一日期格式檢查器。PaddleOCR
使用單執行緒WASM，以配合沒有COOP/COEP標頭的GitHub Pages及降低iPhone記憶體壓力。
日期格式檢查器亦支援包含標題文字的空格格式，例如`BEST BEFORE 31 07 26`。

若生肉與即食食品外觀相似，可在辨認前選擇範圍。辨認即食雞胸時，建議選擇
「即食及冷凍食品」，系統只會比較該類8項候選食品，不會再把生雞胸列為結果。

CLIP模型會在網頁開啟後自動下載及載入，不再顯示手動載入按鈕。使用者可以在模型
載入期間先選擇照片；當模型及照片都準備好後，「開始辨認」按鈕會自動啟用。

## 在電腦測試

```bash
cd clip_food_html
python3 -m http.server 8000
```

在 Chrome 開啟 <http://127.0.0.1:8000>。第一次使用需要連線下載 Transformers.js、
量化 CLIP 模型、Tesseract.js及英文OCR資料，之後瀏覽器通常會快取相關檔案。

## 放到 GitHub Pages

把 `index.html` 和 `.nojekyll` 上載到儲存庫，然後在 GitHub Pages 發佈 `main`
分支的根目錄。用 iPhone 開啟 GitHub Pages 網址後，可拍照或從相簿選擇食物照片。

注意：CLIP 的相似度不能判斷食物是否安全或新鮮；OCR也可能把製造日期誤認為到期日期。
使用者必須確認食品名稱、日期種類及日期內容。
